# A+ Domain Summaries

## What

This is the boiled-down material layer for the complete A+ sections.

Each section explains what a new technician should understand before opening scenario modules.

## Why

The public app should stay practical. These summaries give enough context without becoming a textbook.

## Core 1: Mobile Devices

Mobile support is about small, portable systems that break in predictable ways: power, screens, ports, batteries, wireless, sync, and account access.

A technician should know the difference between a device fault, an accessory fault, a network problem, and an account/service problem. Start with safe external checks: charger, cable, dock, Wi-Fi, Bluetooth, display brightness, storage, and update status.

Do not open sealed devices unless trained and authorised. For many mobile issues, safe support means isolate, document, back up, and escalate.

References: see [reference-links.md](reference-links.md#mobile-devices).

## Core 1: Networking

Networking support is about tracing the path from device to service.

For a helpdesk technician, the key path is: device IP settings, gateway, DNS, local network, internet path, and target service. Wi-Fi connected does not prove internet access. A browser timeout does not prove the website is down.

Good network notes record the SSID, IP address, gateway, DNS server, ping results, lookup results, and whether other users/devices are affected.

References: see [reference-links.md](reference-links.md#networking).

## Core 1: Hardware

Hardware support is about identifying parts, checking safe connections, and knowing when not to touch something.

Most first-line hardware work is external and low-risk: power, cables, ports, monitors, keyboards, mice, drives, printers, and visible error lights. Internal work needs power-off, ESD care, and permission.

A good technician changes one thing at a time and records what was moved, reseated, replaced, or escalated.

References: see [reference-links.md](reference-links.md#hardware).

## Core 1: Virtualization And Cloud Computing

Virtualization means one physical computer can run one or more virtual machines. Cloud means a service runs on someone else's infrastructure and is accessed over a network.

A new technician should understand resource limits: CPU, memory, disk, network, snapshots, and host storage. A slow VM may be a VM problem, a host problem, or a service problem.

Cloud support at A+ level is basic: identify the service, check account access, check service status, and avoid changing tenant-wide settings without approval.

References: see [reference-links.md](reference-links.md#virtualization-and-cloud-computing).

## Core 1: Hardware And Network Troubleshooting

Troubleshooting is a method, not a guess.

Start with the symptom. Reproduce it if safe. Check the easiest likely cause first. Make one change. Test again. If the issue affects many users or shared equipment, escalate before changing infrastructure.

Good troubleshooting notes include symptom, scope, checks, action, result, and next step.

References: see [reference-links.md](reference-links.md#hardware-and-network-troubleshooting).

## Core 2: Operating Systems

Operating system support is about helping the user work safely on Windows, macOS, Linux, or mobile systems.

A new technician should know where settings live, how updates work, how users and permissions work, and which command-line checks are safe. The goal is not to memorise every menu. The goal is to find evidence without damaging data.

Common OS work includes accounts, storage, updates, services, startup apps, drivers, file permissions, and recovery options.

References: see [reference-links.md](reference-links.md#operating-systems).

## Core 2: Security

Security support is about protecting users, devices, data, and accounts while keeping work moving.

A technician should know the safe response to suspicious messages, weak passwords, MFA problems, lost devices, malware symptoms, and permission requests.

Never collect passwords or MFA codes. Verify through trusted paths. Preserve evidence when compromise is possible. Escalate account compromise, malware, data loss, or policy questions.

References: see [reference-links.md](reference-links.md#security).

## Core 2: Software Troubleshooting

Software troubleshooting is about proving whether the problem is the app, the user profile, the device, the network, the update, or the service.

Capture exact errors before repair. Check scope. Protect user data. Use supported repair options first. Do not reinstall business apps or delete app folders unless authorised.

The best software ticket notes include error text, app version, update timing, user scope, repair action, and result.

References: see [reference-links.md](reference-links.md#software-troubleshooting).

## Core 2: Operational Procedures

Operational procedures are the habits that make technical work safe and useful.

This includes documentation, communication, escalation, change control, backups, safety, privacy, and professionalism. A ticket is not done because the technician is done. It is done when the outcome is clear and the next person can understand what happened.

Good procedure protects the user, the business, and the technician.
