# Lab 05: Troubleshooting Scenarios

## What

Run mixed help desk scenarios using the ticket board.

Focus items:

- observing symptoms
- choosing a station
- checking one thing at a time
- documenting clearly
- escalating when needed

## Why

Real support work mixes hardware, networking, OS, software, security, and communication.

Example:

An app crash might be a software issue, an OS update issue, or a user profile issue. The ticket process keeps the learner from guessing wildly.

## How

Use tickets `APLUS-005` and `APLUS-006`.

Steps:

1. Pick one ticket.
2. Move it to Diagnose.
3. Read the visual clue.
4. Choose a station.
5. Run the first safe check.
6. Move it to Fixing or Document.
7. Write the result.
8. Move it to Done.

Success condition:

```text
I can use the same troubleshooting flow on different problem types.
```

Checklist:

- [ ] I picked one ticket.
- [ ] I chose one station.
- [ ] I ran one first safe check.
- [ ] I wrote the result.
- [ ] I moved the ticket to Done.

## Implementation

Materials:

- `datasets/tickets.json`
- `datasets/symptom-map.csv`
- `diagrams/troubleshooting-flow.md`
- lab completion checklist

Ticket board:

```text
New | Diagnose | Fixing | Document | Done
```

Kinesthetic activity:

Move a physical ticket card across the board as work progresses.

Checklist:

- [ ] Use the board.
- [ ] Say the current stage out loud or write it down.
- [ ] Stop after one complete ticket.

## Assumptions

- The learner may need practice with switching problem types.
- The ticket board should make the switch visible.
- The first mixed session should use only one or two tickets.

Checklist:

- [ ] Keep the session small.
- [ ] Add more tickets later.

## Threat/Risk Notes

- Mixed scenarios can cause overload.
- Guessing creates confusing notes.
- Skipped documentation hides the learning.

Checklist:

- [ ] Pause after 5 stuck minutes.
- [ ] Record known facts.
- [ ] Escalate with notes.

## Validation Steps

Evidence to save:

- completed ticket note
- photo or screenshot of the ticket board

Validation:

- [ ] The learner follows the flow without skipping stages.
- [ ] The ticket note includes problem, checks, fix, result, and reflection.
- [ ] The learner can explain what they would do next if unresolved.
