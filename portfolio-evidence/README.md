# Portfolio Evidence

## What

This folder stores finished lab evidence.

Each evidence pack should show one ticket, one action, one result, and one short reflection.

Example:

`sample-networking-ticket/` shows a simulated DNS troubleshooting ticket.

## Why

The lab is easier to show as a portfolio project when each finished activity has a small evidence pack.

The goal is not to make the work look bigger than it is.

The goal is to make the learner action clear.

## How

Use one folder per completed ticket:

```text
portfolio-evidence/
  sample-networking-ticket/
    README.md
    ticket-note.md
    topology-evidence.md
    reflection.md
    evidence-summary.txt
```

Checklist:

- [ ] Use dummy data.
- [ ] Include one ticket note.
- [ ] Include one visual or diagram.
- [ ] Include one reflection.
- [ ] Remove private details before sharing.

## Assumptions

- Evidence packs are for learning and portfolio practice.
- Real screenshots may be added later only after privacy review.
- A short explanation is better than a long report.

Checklist:

- [ ] Keep each evidence pack small.
- [ ] Use plain language.
- [ ] Keep the troubleshooting path visible.

## Threat/Risk Notes

Risk:

Portfolio evidence can accidentally expose private data.

Response:

Use dummy examples first, then review every real screenshot before sharing.

Checklist:

- [ ] Remove names.
- [ ] Remove emails.
- [ ] Remove public IP addresses.
- [ ] Remove serial numbers.
- [ ] Remove passwords and license keys.

## Validation Steps

Before publishing an evidence pack:

- [ ] Can someone understand the problem quickly?
- [ ] Is the learner action visible?
- [ ] Is the result clear?
- [ ] Is private data removed?
- [ ] Does the evidence match the ticket?
